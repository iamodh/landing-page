## WORKFLOW

### `go` - 구현 사이클

1. **Implement**: 마일스톤의 첫 번째 미완료 항목을 구현한다
2. **Type Check**: `npm run typecheck` 실행 (TypeScript 에러 0개)
3. **Lint**: `npm run lint` 실행 (ESLint 에러 0개)
4. **Build**: `npm run build` 실행 (빌드 성공 확인)
5. **Report & Wait**: 결과를 보고하고 사용자 확인을 기다린다

> ⚠️ 이후 자동 진행 금지. 사용자 피드백에 따라 수정하거나 `commit`을 기다린다.

### `commit` - 완료 처리

1. **코드 커밋**: 변경사항을 성격별로 나누어 커밋한다 (스타일, 컴포넌트, 데이터 등)
2. **PLAN 체크**: 완료된 항목에 체크박스 표시 + `- Commits:` 줄에 커밋 해시 추가

### `verify` - 마일스톤 완료 후 수동 검증

마일스톤 완료 시 사용자가 직접 확인할 수 있는 검증 항목을 제안한다.

**제안 범위:**
- `npm run dev` 후 브라우저에서 확인할 URL과 기대 동작
- Chrome DevTools 반응형 모드에서 확인할 뷰포트 크기
- 클릭/스크롤 등 인터랙션 시나리오
- `npm run build` 성공 여부

**규칙:**
- `commit` 직전 또는 직후에 제안
- 해당 마일스톤에서 구현한 기능에 한정
- 복사-붙여넣기로 바로 실행 가능한 형태로 제공

### 명령어

| 명령어     | 동작                                     |
|------------|------------------------------------------|
| `go`       | 구현 사이클 실행 후 사용자 확인 대기     |
| `commit`   | 커밋 및 PLAN.md 체크박스 표시            |
| `status`   | 현재 마일스톤 진행 테이블 + 다음 항목    |
| `verify`   | 마일스톤 완료 후 수동 검증 제안          |
| `refactor` | 빌드 통과 후 리팩토링 제안               |

## CONSTRAINTS

### npm 사용 규칙

- 새로운 패키지를 사용하기 전에 `package.json`에 존재하는지 확인
- 없으면 코드 작성 전에 추가 필요 여부를 먼저 알려줄 것
- 기존 의존성(React, Tailwind)만으로 해결 가능한지 우선 검토

### 구현 규칙

- 한 번에 여러 섹션 구현 금지 — 마일스톤 단위로만 진행
- TECHSPEC.md에 명시되지 않은 기술 스택 도입 금지
- 빌드 통과 전 리팩토링 금지
- 미리 추상화 금지 — 중복이 실제로 발생한 후에 추출한다
- 섹션 구현 시 `npm run dev`에서 브라우저로 verify 가능한 상태로 완료한다

### BRANCH RULES

- main 브랜치에 직접 커밋 금지
- 마일스톤 시작 시 `feat/m{번호}-{설명}` 브랜치를 생성하고 전환한다
- 브랜치 생성/전환/커밋은 에이전트가 수행, **머지와 push는 사용자가 직접 수행**
- 코드 작성 전 반드시 확인:
  1. 현재 브랜치가 작업 대상 마일스톤과 일치하는지
  2. 이전 마일스톤 브랜치가 main에 머지되었고 현재 브랜치에 반영되었는지

## STACK

> 기술 스택 상세는 **TECHSPEC.md § 4.1** 참조

### Coding Conventions

- **들여쓰기:** 2 spaces
- **네이밍:** 컴포넌트 PascalCase, 변수/함수 camelCase, 파일명 PascalCase(컴포넌트) / camelCase(데이터)
- **커밋 메시지:** Conventional Commits 형식 (`feat:`, `style:`, `data:`, `chore:` 등)
- **커밋 분리:** 문서 수정(CLAUDE.md, PLAN.md)과 기능 추가(코드)는 별도 커밋으로 분리
- **컴포넌트 구조:** 데이터는 `src/data/`에, 표현은 `src/components/`에 — 섞지 않는다
- **상태 최소화:** 전역 상태 금지. 필요한 컴포넌트 내 `useState`만 사용
- **Tailwind 우선:** 커스텀 CSS 작성 전 Tailwind 클래스로 해결 가능한지 먼저 확인
